C++11新特性

1. auto和decltype

c++11引入了auto和decltype这两个关键字实现了类型推导，让编译器来操心变量的类型。

1.1. auto

auto用于两种情况

  • 声明变量时根据变量初始值的类型自动为此变量选择匹配的类型。
  • 声明函数时函数返回值的占位符。

例子：

auto a = 3.14;  //double
auto b = new auto(9);  //int*
auto f = [](double x) {return x * x; };  //类型为std::function<double(double)>函数对象

//返回类型后置，auto为返回值占位符
auto sum(int a, int b) -> int {
	return a + b;
}

1.2. decltype

decltype主要用于提取变量和表达式的类型。

语法形式：decltype(e)，e可以是变量、表达式、函数名。

  1. decltype + 变量

当使用decltype(var)的形式时，decltype会直接返回变量的类型（包括顶层const和引用）

const int ci = 0;
const int& cj = ci;
decltype(ci) x = 0;  //const int
decltype(cj) y = x;  //const int&
  1. decltype + 表达式

当使用decltype((expr))的形式时，decltype会返回表达式结果对应的类型。一个表达式的结果不是左值，就是右值。因此，decltype(expr)的结果根据expr的结果不同而不同：expr返回左值，得到该类型的左值引用；expr返回右值，得到该类型。

int a = 10;
decltype(a+10) b = 10;  //int
decltype((a)) c = b; //加了括号，变成了表达式，返回的是i的左值形式，int&
int *p = &a;
decltype(*p) d = b; //int&
  1. decltype + 函数

当使用decltype(func_name)的形式时，decltype会返回对应的函数类型，不会自动转换成相应的函数指针。

int add(int a,int b){
	return a+b;
}
decltype(add) *pf = add;
  1. decltype(auto)

decltype(auto)是C++14新增的类型指示符，可以用来声明变量以及指示函数返回类型。 当decltype(auto)被用于声明变量时，该变量必须立即初始化。

decltype(auto) a = 10;
decltype(auto) add(int a,int b){
	return a+b;
}

remove_reference

引用移除：

//c++11
template<class T> struct remove_reference {typedef T type;};
template<class T> struct remove_reference<T&> {typedef T type;}; //偏特化版本（左值）
template<class T> struct remove_reference<T&&> {typedef T type;}; //偏特化版本（右值）
//c++14
template<class T> using remove_reference_t = typename remove_reference<T>::type;

由于模板参数T&&和T&的存在，它有可能被一个左值引用或右值引用的参数初始化，从而发生类型的变化，这种变化就称为引用折叠。

引用折叠规则：

& + & 折叠成 &

& + && 折叠成 &

&& + & 折叠成 &

&& + && 折叠成 &&

如果任一引用为左值引用，则结果为左值引用。否则（即两个都是右值引用），结果为右值引用。

当将一个左值传递给一个参数是右值引用的函数，且此右值引用指向模板类型参数(T&&)时，编译器推断模板参数类型为实参的左值引用。

注意：使用remove_reference_t&作为函数参数类型时，只能接受参数为左值的调用。

std::move的使用

std::move无条件将它的参数强制转化为右值引用

template <class _Ty>
_NODISCARD constexpr remove_reference_t<_Ty>&& move(_Ty&& _Arg) noexcept { // forward _Arg as movable
    return static_cast<remove_reference_t<_Ty>&&>(_Arg);
}

首先，函数参数T&&是一个指向模板类型参数的右值引用，通过引用折叠，此参数可以与任何类型的实参匹配（可以传递左值或右值）。

简单来说，右值经过T&&传递类型保持不变还是右值；而左值经过T&&变为普通的左值引用。

std::forward的使用

std::forward实现了参数在传递过程中保持其值属性的功能，即若是左值引用，转换后仍是左值引用；若是右值引用，则转换之后仍然是右值引用。

// FUNCTION TEMPLATE forward
template <class _Ty>
_NODISCARD constexpr _Ty&& forward(
    remove_reference_t<_Ty>& _Arg) noexcept { // forward an lvalue as either an lvalue or an rvalue
    return static_cast<_Ty&&>(_Arg);
}

template <class _Ty>
_NODISCARD constexpr _Ty&& forward(remove_reference_t<_Ty>&& _Arg) noexcept { // forward an rvalue as an rvalue
    static_assert(!is_lvalue_reference_v<_Ty>, "bad forward call");
    return static_cast<_Ty&&>(_Arg);
}

std::forward主要用于实现完美转发的功能。

void func(int& arg) {
	cout << "func lvalue" << endl;
}

void func(int&& arg) {
	cout << "func rvalue" << endl;
}

template<typename T>
void wrapper(T&& args) {
	func(args);
    //func(std::forward<T>(args));
}

int main()
{
	int a = 10;
	wrapper(a);
	wrapper(10);

	return 0;
}

程序输出的结果是：

func lvalue
func lvalue

根本原因其实是，右值引用属性不能被转发。所以，c++11提供了std::forward()函数用于完美转发。在转发过程中，左值引用在被转发后仍然保持左值属性，右值引用在被转发后依然保持右值属性。

可变参数模板

C++11增强了模板功能，允许模板定义中包含0到任意个模板参数。

可变参数模板和普通模板的语义是一样的，只是写法上稍有区别，声明可变参数模板时需要在typename或class后面带上省略号...

// T叫模板参数包，args叫函数参数包
template<class ... T>   
void func1(T ... args) {
}
  1. 通过递归方式展开参数包

需要提供一个参数包展开的函数和一个递归终止函数。下面是通过可变参数模板以及递归展开实现的一个打印函数：

#include <iostream>
#include <stdexcept>
using namespace std;

void Debug(const char* s)
{
    while (*s)
    {
        if (*s == '%' && *++s != '%')
        {
            throw runtime_error("invalid format string: missing arguments");
        }
        cout << *s++;
    }
}

template<typename T, typename... Args>
void Debug(const char* s, T value, Args... args)
{
    while (*s)
    {
        if (*s == '%' && *++s != '%')
        {
            cout << value;
            return Debug(++s, args...);
        }
        cout << *s++;
    }
    throw runtime_error("extra arguments provided to Debug");
}

int main()
{
    Debug("a = %d, b = %c, c = %s\n", 250, 'm', "mike");
    return 0;
}
  1. 非递归方式展开：
#include <iostream>
using namespace std;

template <class T>
void print(T arg)
{
    cout << arg << endl;
}

template <class ... Args>
void expand(Args ... args)
{
    int a[] = { (print(args), 0)... };
}

int main()
{
    expand(1, 2, 3, 4);
    return 0;
}

逗号表达式的值相当于最后一项的值; 逗号表达式(print(args), 0),会先执行print(args),再得到逗号表达式的结果0 同时，通过初始化列表来初始化一个变长数组，{(print(args) , 0)...}将会展开成(print(arg1), 0), (print(arg2), 0).....最终会创建一个元素都为0的数组int a[sizeof...(args)]。

线程同步与互斥

std::mutex是c++11中最基本的互斥量，一个线程将mutex锁住时，其它的线程就不能操作mutex，直到这个线程将mutex解锁。

void lock()		//加锁
void unlock()	//解锁
bool try_lock()	//尝试加锁

mutex很好地解决了多线程资源争抢的问题，但也存在缺点，加锁和解锁需要消耗一定的时间，如果线程中加锁解锁的次数太多，会导致程序很慢。这时候就可以使用std::atomic。 用atomic<T>来定义共享的资源，多线程可以同步地去操作atomic对象，从而省去了mutex上锁、解锁的时间消耗。

enable_shared_from_this使用

原型：template< class T > class enable_shared_from_this;

使用场合：当类A被share_ptr管理，且在类A的成员函数里需要把当前类对象作为参数传给其他函数时，就需要传递一个指向自身的share_ptr。

1.为何不直接传递this指针

使用智能指针的初衷就是为了方便资源管理，如果在某些地方使用智能指针，某些地方使用原始指针，很容易破坏智能指针的语义，从而产生各种错误。

2.可以直接传递share_ptr<this>么？

答案是不能，因为这样会造成2个非共享的share_ptr指向同一个对象，未增加引用计数导对象被析构两次

#include <iostream>
#include <memory>

class Test
{
public:
	auto getPtr() {
		return std::shared_ptr<Test>(this);
	}
	~Test() {
		std::cout << "~Test()" << std::endl;
	}
};

int main()
{
    // 错误的示例，每个shared_ptr都认为自己是对象仅有的所有者
	std::shared_ptr<Test>s1(new Test());
	std::shared_ptr<Test>s2 = s1->getPtr();
	std::cout << s1.use_count() << std::endl;
	std::cout << s2.use_count() << std::endl;
	return 0;
}


//输出
/*
1
1
~Test()
~Test()
*/

正确的实现：

#include <iostream>
#include <memory>

class Test : public std::enable_shared_from_this<Test>
{
public:
	auto getPtr() {
		return shared_from_this();
	}
	~Test() {
		std::cout << "~Test()" << std::endl;
	}
};

int main()
{
	// 错误的示例，每个shared_ptr都认为自己是对象仅有的所有者
	std::shared_ptr<Test>s1(new Test());
	std::shared_ptr<Test>s2 = s1->getPtr();
	std::cout << s1.use_count() << std::endl;
	std::cout << s2.use_count() << std::endl;
	return 0;
}

//输出
/*
2
2
~Test()
*/

c++普通函数在头文件定义报重复定义的错误。而class定义不会

类的定义不产生目标代码，只是告诉编译器，类的数据格式是如何的，实例化后对象该占多大空间，它和普通变量的声明唯一的区别是不能在同一编译单元内出现多次。同一编译单元内，类重复定义，会编译时报错,因为编译器不知道在该编译单元中，实例化对象的时候该调用哪一个定义的类？但是在不同编译单元内，类可以重复定义,因为类的定义未产生实际代码。因为它具有内部链接，（所谓内部链接指的是该名称对于所在编译单元是局部的，在链接时不会与其他编译单元中同样 的名称产生命名冲突）,所以类定义在头文件中在链接时候不会冲突。
